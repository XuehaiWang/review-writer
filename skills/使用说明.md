# 使用说明

本说明描述如何用通用版综述写作技能包）从零开始生成一篇文献综述，适用于任意学科方向，不限于化学。整体流程分三步：

1. 把 20 篇以上 PDF 论文放进知识库文件夹
2. 调用相关技能，把这些论文解析并编码成结构化 metadata
3. 运行 `review-writing-orchestrator`，走完从选题到导出 Word 的全部阶段

## 第一步：上传论文到知识库

网站内有一个知识库（http://223.109.239.32:14388/sci/knowledge/user-literature），可以批量上传文件，使用中请只上传PDF论文文件。

操作：

1. 建议一次性放入 **20-30 篇** 与选题相关的 PDF（可以掺入少量不太相关的论文作为噪声，测试筛选效果）。更大的论文库（50 篇以上）也支持，但请按下文"大型知识库与 token 控制"的两遍式流程走。
2. 文件名不需要特殊格式，Agent 会自动读取标题、作者等信息。

上传完成后，直接告诉 Agent「论文已经放好了」，不需要手动指定路径——后续技能会自己去找。

## 第二步：调用技能构建 metadata

这一步分两个子阶段：**解析 PDF** 和 **编码 metadata**。

### 2.1 解析 PDF（MinerU）

调用 `mineru-precise-parse-review-writer` 技能，把知识库里的 PDF 转成 Markdown、图片、`content_list.json` 等结构化产物：

```text
请使用 mineru-precise-parse-review-writer 技能解析知识库里的论文
```

Agent 会自己定位知识库目录并输出到 `<review-root>/mineru-outputs/`。如果论文已经解析过，可以跳过这一步。

### 2.2 编码 metadata（结构化标签）

调用 `review-metadata-prep` 技能。这一步不需要你手动传路径——**Agent 会自己去找知识库文件夹**。

**小型知识库（约 30 篇以内）**，直接全库 LLM 编码：

```text
请使用 review-metadata-prep 技能，用 LLM 模式给知识库里的论文编码 metadata
```

**大型知识库（超过 30 篇，如 50 篇以上）**，全程自动两遍式编码（tag-after-discovery），不需要你做任何设置：

- 第二步照常调用（同上面那句话即可）——论文数超过阈值时，技能会自动先只做规则抽取并给出提示；
- 第三步 discovery 阶段，**你确认候选集之后，Agent 会自动只给入选论文补打 LLM 标签**，不需要你再说什么。

如果确实想给整个大库全部打标签（费 token），明确告诉 Agent「全库强制 LLM 打标签」即可。若没有可用的后端 API Key，规则模式之后也可以直接让 Agent 阅读论文首页手动填补标签（仅建议小库）。LLM/Agent 会为每篇论文提取：

```text
output           主要产出/结果/成果
input            主要研究对象/起始材料
method           核心方法/技术/路线
co_input         协同投入物（多数论文不适用，留空是正常的）
modifier         修饰/调控要素（多数论文不适用，留空是正常的）
process_type     过程/研究类型
document_scope   文献类型（研究论文/综述/纪要等）
```

编码完成后，可以跑一次校验：

```text
请使用 review-metadata-prep 技能校验 metadata
```

确认没有阻断性问题（缺 `paper_id`、缺摘要、JSON 格式错误等）之后再进入第三步。

## 第三步：运行 review-writing-orchestrator

调用总控技能，Agent 会依次跑完全部 10 个阶段，每个阶段结束后停下来等你确认（结论生成、总结图两个阶段由 Agent 自动完成，无需额外确认）：

```text
请使用 review-writing-orchestrator 技能，围绕 <你的选题> 生成一篇综述
```

Orchestrator 会自动：

1. 把非英文选题翻译成英文（检索用英文关键词更准）。
2. 生成 `project_id` 并在 `review-projects/<project_id>/` 下建项目文件夹。
3. 依次调用以下 10 个子技能：

| 阶段 | 技能 | 你需要确认什么 |
| --- | --- | --- |
| 1 | review-topic-paper-discovery | 20-30 篇候选论文是否合适；**同时查看报告里的 Borderline Papers（临界论文）名单**，把其中明显贴题的论文提回候选集 |
| 2 | review-literature-matrix-outline | 文献矩阵与 `selected_outline.md` |
| 3 | review-section-blueprint | 章节/段落/论文/图表映射关系 |
| 4 | review-section-drafting-figure-picking | 各章节草稿与配图候选 |
| 5 | review-figure-style-redraw | 重绘图表是否可用 | **该步骤由于缺少API Key，建议跳过**
| 6 | review-draft-merge-polish | 合并后的初稿 |
| 7 | review-conclusion-generator | 无需额外确认（Agent 自动生成结论/挑战/展望章节并自校验） |
| 8 | review-final-audit-release | 终稿内容与格式审查 |
| 9 | review-outline-summary-chart | 无需额外确认（Agent 自动生成全文与分章节结构总结图） |
| 10 | review-export-docx | 下载 `final_draft.docx` 并在 Word 里核对样式 |

**每个阶段跑完后，Agent 会停下来等你确认**，除非你明确说「不用停，一次跑完」之类的话。想了解整体进度时，直接问 Agent：

```text
请用 review-writing-orchestrator 技能查看当前项目进度
```

阶段 1（discovery）内部有两个新机制，Agent 会自动执行、你只需要知道其含义：

- **词表对齐扩展**：扩展检索关键词之前，Agent 会先读取知识库现有的标签/标题词表，并把库内实际用词加入关键词，避免"论文贴题但措辞不同"导致的漏选。
- **临界论文复核**：打分略低于入选线的论文不会被静默丢弃，而是列入 `discovery_report.md` 的 "Borderline Papers" 名单，由 Agent（和你）逐条判断是否提回候选集。

## 大型知识库与 token 控制

当知识库论文较多（如 50 篇以上）或后端 API 按 token 计费时，按下面的组合使用：

1. **两遍式编码（自动）**：论文数超过 30 时 `review-metadata-prep` 会自动先做规则建库，discovery 选出候选集后再只给入选论文做 LLM 打标签，成本从"全库"降到"候选集"（见 2.2）。
2. **相关性预过滤**：如果明确要求全库 LLM 编码，可以同时告诉 Agent「先按检索关键词做相关性预过滤，离题论文跳过 LLM」——跳过原因会记录在 metadata 里，可审计。
3. **脚本化文献矩阵**：阶段 2 时告诉 Agent「用脚本化方式构建文献矩阵」——每篇论文一次独立、可断点续跑的后端 API 调用，避免 Agent 在长对话里反复携带全部上下文读论文；跑完后 Agent 只需抽查 2-3 行。
4. **分级用模型**：可以告诉 Agent 机械抽取类工作（打标签、矩阵行）用便宜的小模型，把大模型留给大纲选择、蓝图、章节撰写与终稿审查。

## 常见追加需求

- **翻译成中文**：终稿确认后，让 Agent 把 `07_final_audit/final_draft.md` 翻译成中文，输出到同目录下的 `final_draft_zh.md`。
- **导出中文 Word**：翻译完成后，再次调用 `review-export-docx`，指定 `--input` 为 `final_draft_zh.md`，输出到 `09_docx_export/final_draft_zh.docx`。
- **扩大论文池**：如果发现候选论文不够多或不够贴题，可以随时往知识库文件夹里加论文，重新跑第二步编码，再回到 `review-topic-paper-discovery` 重新筛选。
