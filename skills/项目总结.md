# review-writer 项目总结

## 1. 项目概念与定位

`review-writer` 是一套**领域无关（domain-agnostic）的文献综述自动写作技能包**，目标是把"从一堆 PDF 论文到一篇带正式引用格式、可导出 Word 的综述"这一完整流程，拆解成 8 个可独立调用、可人工检查点插入的技能（skill），每个技能对应流程中的一个阶段。

## 2. 整体结构与工作流

### 2.1 目录结构

```text
<review-root>/                          # 项目根目录
  skills_versa/                         # 本技能包（11 个 skill + 1 个模板目录）
  skills/                               # 化学特化原版（不要动，除非明确要求）
  review-library/                       # 论文知识库
    paper_pdf/                          # 原始 PDF（或由 --pdf-root 指定的其他目录）
    metadata/papers/<paper_id>.metadata.json
    registry/papers.jsonl
  mineru-outputs/                       # MinerU 解析产物（Markdown、图片、content_list.json）
  review-projects/<project_id>/         # 每个综述项目一个文件夹，内部按阶段分子文件夹
    00_discovery/
    01_matrix_outline/
    02_section_blueprint/
    03_section_drafting/
    04_figure_redraw/
    05_first_draft/
    06_final_audit/
    07_docx_export/
```

**8 阶段 = 8 个独立文件夹**（本会话中完成的重构）：每个阶段对应一个数字前缀文件夹，避免早期版本"两个阶段共用一个文件夹"的混乱问题。仅 `skills_versa/` 使用此编号；已存在的旧项目（如 `urban-heat-island-mitigation`、`test-allene`）仍用旧的 6 文件夹结构，未做迁移。

### 2.2 工作流（由 `review-writing-orchestrator` 统一调度）

```text
1. review-topic-paper-discovery       主题 → 候选论文（本地库打分 + Crossref 网络检索）
2. review-literature-matrix-outline   已选论文 → 文献矩阵 + 大纲选项 → 人工选定 selected_outline.md
3. review-section-blueprint           大纲 → 各章节的论文分配、论点、图表需求
4. review-section-drafting-figure-picking  逐章节撰写正文 + 挑选真实配图
5. review-figure-style-redraw         （可选，需图像生成 API）统一风格重绘配图
6. review-draft-merge-polish          合并各章 → 引用编号 [1][2]... → ACS 格式参考文献列表
7. review-final-audit-release         内容/格式终审 → 定稿 final_draft.md
8. review-export-docx                 定稿 → final_draft.docx
```

每个阶段跑完后**默认停下来等待人工确认**（除非用户明确要求连续跑完），这是整个流程设计上的核心原则——写作过程始终保留人工把关点。

### 2.3 配套的元数据准备技能

在跑上述 8 阶段之前，还有一个前置技能负责论文入库：

```text
mineru-precise-parse-review-writer   PDF → Markdown/图片/content_list.json（MinerU 解析）
review-metadata-prep                 解析产物 → 结构化 metadata.json（含开放词表标签）
```

## 3. 各 Skill 简介

| Skill | 作用 | 关键脚本 |
|---|---|---|
| `mineru-precise-parse-review-writer` | 调用 MinerU 把 PDF 解析成 Markdown、图片、`content_list.json` | `parse_review_writer_pdfs.py` |
| `review-metadata-prep` | 从解析产物提取标题/作者/年份/期刊/卷/页等文献信息，并用 LLM 打上 7 类开放词表标签（`output/input/method/co_input/modifier/process_type/document_scope`）；支持相关性预过滤、Crossref 补全、批量续跑打标签 | `prepare_metadata.py`、`crossref_backfill_metadata.py`、`batch_llm_retag_metadata.py` |
| `review-topic-paper-discovery` | 关键词扩展 + 本地论文打分 + 可选 Crossref 网络检索，产出候选论文集，含"临界论文"复核机制 | `discover.py` |
| `review-literature-matrix-outline` | 逐篇精读已选论文，产出统一字段的文献矩阵（含约 1000 词的 `main_content` 摘要）与 2-3 个大纲选项 | 无脚本，纯 Agent 阅读撰写 |
| `review-section-blueprint` | 把选定大纲转成结构化的章节蓝图：每章的论文分配、论点、字数目标；支持从大纲中读取显式 `Papers: P001, P002` 分配（避免关键词打分在词汇高度重叠时分配错乱） | `init_section_blueprint.py` |
| `review-section-drafting-figure-picking` | 逐章节撰写正文，并从真实 MinerU 抽取的图表中挑选配图；先构建共享的 `paper_content_cache.json` 供各章节子任务复用，避免重复整篇重读论文 | `build_paper_content_cache.py`、`build_paper_figure_inventory.py`、`select_initial_figure_candidates.py` |
| `review-figure-style-redraw` | （可选）用图像生成 API 把源图重绘成统一学术风格；缺 API Key 时整个阶段可跳过 | 无独立脚本示例 |
| `review-draft-merge-polish` | 合并各章节草稿为完整初稿；把正文中的 `paper_id` 引用转换为全局唯一的 `[1][2]...` 编号；插入配图；生成 ACS 期刊格式的参考文献列表 | `init_first_draft.py`、`insert_figures_into_draft.py` |
| `review-final-audit-release` | 逐章节做内容审查（证据支撑、引用准确性、过度断言）+ 一次性的格式扫描（标题层级、引用编号完整性、参考文献格式），产出定稿 | `final_audit_scan.py` |
| `review-export-docx` | 把 Markdown 定稿转换成符合期刊模板样式的 Word 文档（标题/摘要/图注/参考文献等样式映射） | `md2docx.py` |
| `review-writing-orchestrator` | 总控技能，负责选题翻译、项目命名、依次调用以上各阶段、展示当前进度 | `project_status.py` |

## 4. 实际使用方法与规范

### 4.1 最简单的完整调用方式

```text
"请使用 review-writing-orchestrator 技能，围绕 <你的选题> 生成一篇综述"
```

Orchestrator 会自动把选题翻译成英文、生成 `project_id`、依次调用 8 个子技能，每阶段结束后停下等待确认。查看进度：

```text
"请用 review-writing-orchestrator 技能查看当前项目进度"
```

### 4.2 论文入库（跑综述之前）

1. 把 PDF 放进知识库文件夹（推荐 `review-library/paper_pdf/`，但任何路径都可以，Agent 会自动定位，或用户可指定）。
2. 调用 `mineru-precise-parse-review-writer` 解析。
3. 调用 `review-metadata-prep`（建议加 `--use-llm`），给论文打开放词表标签；论文数超过 30 篇时建议用相关性预过滤（`--relevance-keywords`）控制成本。
4. 可选：跑 `crossref_backfill_metadata.py` 补全期刊/卷/页信息。

### 4.3 关键规范与约定

- **`skills/` 与 `skills_versa/` 严格分离**：除非用户明确要求，绝不修改 `skills/`；只有确认是"与学科无关的纯 bug"（路径处理、编码问题等）才考虑同步到两边。
- **人工检查点是硬性要求**：每阶段跑完必须停下来，明确告知用户应该检查什么，除非用户明确说"不用停，一次跑完"。
- **引用格式（ACS 期刊风格）**：正文引用用全局唯一的 `[N]` 编号（按首次出现顺序分配，非按大纲顺序）；参考文献格式为 `[N] Last, F.; Last, F. Title. *Journal* **Year**, *Volume*, Pages.`（期刊斜体、年份加粗、卷号斜体、页码纯文本，无年份前逗号）；缺失字段（期刊/卷/页）按规则优雅省略，不打印占位符文字。
- **图注来自真实 MinerU 抽取的图片**，不得凭空编造；重绘前的图统一标注"redraw still required before final release"。
- **成本控制是设计原则之一**：文献矩阵阶段限制单篇论文读取长度并分批处理；章节撰写阶段构建共享内容缓存避免重复读全文；终审阶段逐章节审查而非一次性加载全部证据；metadata 打标签阶段支持相关性预过滤，避免给明显跑题的论文（如噪声测试论文）付费打标签。
- **开放词表标签**：7 类标签（`output/input/method/co_input/modifier/process_type/document_scope`）由 LLM 自由撰写具体标签值，而非从固定枚举中选择——这是保证跨学科通用性的核心设计。其中 `co_input` 和 `modifier` 多数论文不适用，留空（`not specified`）是正常现象，不应视为质量问题。
- **文件夹产出必须写完整**：每个阶段有固定的必需输出文件清单（见各 skill 的 `SKILL.md` "Outputs" 部分），`review-writing-orchestrator` 靠这些文件的存在与否判断阶段是否完成。
